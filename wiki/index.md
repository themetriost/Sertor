# Wiki di Produzione — Sertor

Wiki della **fase di produzione** (costruzione del CLI `sertor`, vedi
[`../requirements/sertor-cli/epic.md`](../requirements/sertor-cli/epic.md)). È **nuovo** e
cumulativo: cresce a ogni sessione secondo lo schema in [`../CLAUDE.md`](../CLAUDE.md)
(sezione *Wiki & documentazione*), ispirato al pattern "LLM Wiki" di Karpathy.

> **Il wiki del prototipo è altrove e congelato.** I 4 motori RAG su FastAPI, il loro codice e il
> wiki storico vivono ora in [`../prototype/`](../prototype/) come **snapshot di sola lettura**.
> Quel wiki (`../prototype/wiki/`) è indicizzato nel **RAG di dogfooding**: per consultare il
> prototipo si interroga il server MCP **`sertor-rag`** (tool `search_code` / `search_docs` /
> `get_context` / `find_symbol` / …), non si modifica più a mano.

### Concepts (fondamenta e astrazioni)

- **[[retrieval-core]]** — Il **nucleo di retrieval** importabile (`sertor-core`), *il prodotto*: architettura Clean (domain/services/adapters/engines + porte `Protocol`), composition root guidato da `Settings`, backend `local`/`azure`, policy errori tollerante↔strict, collezioni namespaced per `(corpus, provider)`. CLI/MCP/wiki ne sono consumatori sottili. *Scomposto nelle 4 pagine-entità sotto.*
- **[[retrieval-vs-graph]]** — Guida di scelta: ricerca **ibrida** (scopri per significato) vs **code-graph** (naviga per struttura), pattern *scopri → naviga*, tabella domanda→tool. Ortogonali, non alternative.
- **[[domain-model]]** — Le **entità dati pure** del nucleo (`Document`, `Chunk`/`ChunkMetadata`, `EmbeddedChunk`, `RetrievalResult`, `IndexReport`; enum `DocType`/`ChunkerKind`): nessun SDK nel dominio, id stabili (path POSIX, `doc_id#index`) → idempotenza del rebuild.
- **[[ports-adapters]]** — Le **sei porte** `Protocol` (`EmbeddingProvider`, `VectorStore`, `LexicalIndex`, `Reranker`, `CodeGraph`, `RetrieverStrategy`) e gli **adapter** che le implementano (Ollama/Azure · Chroma/Azure Search · BM25 · FlashRank · networkx); il composition root sceglie da `Settings` con import lazy. Structural typing → mockabili.
- **[[chunking-dispatch]]** — Il **chunking** `Document`→`Chunk`: dispatch per tipo/lingua (markdown / sintattico tree-sitter / fallback dimensionale), 10 linguaggi sintattici, esclusione deliberata R-N2 di PowerShell/SQL, id stabile `doc_id#index`.
- **[[indexing-and-retrieval]]** — Le **due pipeline**: indicizzazione (ingest→chunk→embed→store, atomicità del rebuild, DEFAULT incrementale da FEAT-009 con manifest persistito) e la **facade** `search_code/docs/combined`, tollerante su indice assente (`[]`+warning). Punto d'ingresso dei consumatori via `build_facade()`.
- **[[vector-retrieval]]** — La **prima modalità RAG**: retrieval vettoriale (embed query → similarity top-k) realizzato dal motore baseline; policy errore *strict* (`IndexNotFoundError`) + valutazione hit-rate@k/MRR@10. Dal 2026-06-12 non è più il default (→ [[hybrid-retrieval]]) ma resta la baseline di confronto, selezionabile con `SERTOR_ENGINE=baseline`.
- **[[hybrid-retrieval]]** — La **seconda modalità RAG e nuovo default** (FEAT-004, PR #24): BM25 lessicale (porta `LexicalIndex`, sidecar atomico) + via densa fusi con RRF deterministico; degradazione onesta sui corpora pre-ibrido (warning, mai errore); reranking FlashRank come extra opzionale; consumatori invariati via strategia iniettata. Qualità: simboli hit@5 0→1.00, xfail storici chiusi strict.
- **[[retrieval-confidence]]** — Il **segnale di confidenza** (FEAT-018, PR #32): soglia di similarità opzionale (`SERTOR_MIN_SCORE`, default off) che esclude i risultati deboli ed emette `low_confidence`, dando all'agente il materiale per **astenersi** (grounding nel modello composito); additivo (contratto invariato). Asimmetria voluta: nell'ibrido filtra il pool denso **prima di RRF** (lo score RRF non è una similarità). Compagno: retry resilienza embedder (stessa 018).
- **[[valutazione-e-non-regressione]]** — La **capacità di valutazione del retrieval** (FEAT-001, epica retrieval-qualita): ground-truth versionato su disco, `sertor-rag eval` per hit-rate@k/MRR con gate di non-regressione su baseline+tolleranza, skill di genesi assistita e feedback pertinenza; ciclo di vita completo suite di valutazione host-side.
- **[[code-graph]]** — La **navigazione strutturale del codice** (FEAT-005, PR #25), ortogonale ai motori: grafo AST deterministico (porta `CodeGraph`, artefatto JSON per corpus, build dentro `index()` — mai stantio), copertura per-linguaggio DICHIARATA e verificata sui 10 linguaggi; i 4 tool MCP storici tornati (find_symbol/who_calls/related_docs/get_context); due semantiche di assenza, niente LLM/cloud.
- **[[thin-consumer]]** — Il pattern per cui le interfacce (CLI, server MCP, tool) espongono il [[retrieval-core]] importandolo e cablandolo dalle factory `build_*`, **senza reimplementare logica**: il prodotto è la libreria, l'interfaccia è un guscio sottile (host-agnostico, Principio X). Esempio realizzato: il server MCP.
- **[[speclift]]** — Pipeline **deterministica diff→requisiti ancorati** (Sinthari): 9 stadi meccanici (ingest→parse→filter→locate→bundle→verify→render) + giudizio agentico. Il moat riverifica ogni àncora sul filesystem. Due adapter locator: Adapter A via CLI Sertor, Adapter B via MCP (adatto al consumo esterno). Artefatto: `located.json` con field `verified`. Integrazione Sertor: three-gear agente-driven via MCP.

È la metà **D** del confine D↔N.
- **[[mcp-server]]** — Il server MCP `sertor-rag` (`sertor_mcp`): la superficie che espone la facade del core come 3 tool (`search_code/docs/combined`), facade memoizzata, formato citabile `path#chunk`, trasporto stdio. Esempio canonico di [[thin-consumer]].
- **[[sertor-rag-cli]]** — La CLI di esecuzione RAG `sertor-rag` (feature 011, PR #21): `index`/`search` dal terminale, exit code per scripting, anteprime troncate (`preview_chars`), osservabilità `-v`/`--log-json`/`--log-config`, validazione statica del backend (`validate_backend`). Secondo esempio realizzato di [[thin-consumer]]; comandi spartiti per DA-8 (`sertor`=installer, core=esecuzione).
- **[[sertor-install-kit]]** — ✨ **Toolkit di installazione estratto** (feature 037, 2026-06-15): motore generico, stdlib-only, zero dipendenza dal core. Riusato da [[sertor-installer]] (wiki+rag) e [[sertor-flow]] (governance). Tassonomia `Artifact`/`ArtifactKind`, merge intelligenti, executor fail-fast, logging strutturato, errors. NFR-2 (DRY).
- **[[sertor-installer]]** — Il pacchetto/comando installer `sertor` (feature 012, PR #22): `sertor install wiki` porta il sistema-wiki su un ospite (skill+agente+hook+rituale a marker+config inferita+struttura), non distruttivo per artefatto, idempotente, install≠run; assets **package-data** come fonte canonica (`.claude/` del repo = derivato + test di guardia). Motore ora in [[sertor-install-kit]]. Aperto: tema lingua degli asset.
- **[[sertor-flow]]** — ✨ **Installer di governance/SDLC** (feature 037, 2026-06-15): pacchetto separato che porta metodo SDLC (SpecKit vendor, costituzione-starter, rituale) su un ospite **senza dipendere dal RAG** (REQ-002). Riusa [[sertor-install-kit]], thin consumer, host-agnostico (Principio X).
- **[[assistant-targeting]]** — ✨ **Targeting per-assistente** (FEAT-007, feature 044, PR #64, 2026-06-15): il seam `AssistantId`/`Surface`/`AssistantProfile` nel [[sertor-install-kit]] che rende l'installer capace di depositare le superfici per **Claude e GitHub Copilot** a parità funzionale. Decisione DA-2 = ibrido (riuso contenuto + traduzione contenitore, fonte unica + guardia anti-drift); plan-builder parametrici, `merge_mcp` con `root_key`. Embodiment del Principio X all'assistente. Riuso da FEAT-009.
- **[[installer-lifecycle]]** — ✨ **Ciclo di vita dell'installer** (FEAT-008, feature 048, PR #71, 2026-06-17): i verbi `upgrade`/`uninstall` per `sertor` e `sertor-flow` oltre al primo install. Verbo ortogonale `LifecycleOp` + funzioni inverse pure **una volta sola nel kit** (no raddoppio tassonomia, divergenza impossibile per costruzione), riuso degli stessi plan-builder col verbo, diff a posteriori `sertor_owned_paths` + test invariante `plan ⊆ owned` (no manifest), `--purge-wiki` opt-in CI-safe, report `install.report/1` esteso additivo, tipi artefatto A/B/C/D. Invariante: `sertor-flow` senza dipendenza da `sertor-core`/`sertor`.
