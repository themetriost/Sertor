## [2026-06-24] record | E12-FEAT-010 follow-up — uv run --project (non --directory): fix indicizzazione cwd

Un agente ospite su Windows invocava `uv run --directory .sertor sertor-rag index .`, che faceva indicizzare il contenuto della **cartella `.sertor/` anziché la radice del progetto**, perché il flag `--directory` cambia la working directory del sottoprocesso. La conseguenza: l'indice restava vuoto (documents=0).

- **Bug:** `uv run --directory` cambia il cwd; i path relativi come `index .` risultano risolti in `.sertor/` → indicizzazione fallace. Footgun su ogni comando col cwd-relativo.
- **Causa-radice verificata:** `settings.py` ancora corretto (index dir da `.sertor/.env`, ancorato a `.sertor/.index`); il danno era nel **comportamento dell'ospite**, non nel core.
- **Fix:** sostituire `uv run --directory .sertor` con **`uv run --project .sertor`** in tutti gli asset agent-facing + docs + hook. Il flag `--project` usa il venv di `.sertor` **senza cambiare il cwd** → i path relativi tornano corretti; `index .` = radice; `index ..` diventa ridondante → `index .`
- **Monito anti-footgun:** nuovo blocco «How to invoke» ribadisce il **perché** («`--project` NOT `--directory`») così la trappola non si ripete su una copia/adattamento ospite.
- **Componenti invariati:** template server MCP (usa `--directory`, scope diverso — genera comandi per llama.cpp/ollama), core, kit.
- **Esiti:** 65 conversioni su 17 file; sertor 372 + kit 131 test verdi; ruff clean; parità Copilot integra; verification empirica live su VM reale.

## [2026-06-24] record | E10-FEAT-011 dogfooding discovery — stdin-hang in rag-freshness.ps1 hook

Testing the newly implemented FEAT-011 enforcement hook (`rag-freshness.ps1`) by manual invocation revealed a footgun: the script blocks indefinitely on `[Console]::In.ReadToEnd()` when stdin is not redirected, because the hook reads its payload from stdin before invoking `sertor-rag index`. When run without piping (interactive terminal), it never reached the re-index command; the process remained hung for ~1 hour waiting on stdin.

- **Symptom:** manual invocation `& .\.claude\hooks\rag-freshness.ps1` appended to the process; no re-index occurred; Chroma index mtime and `.sertor/.rag-health.json` were not written.
- **Diagnosis:** hook's docstring claimed «tolerant if absent (manual invocation)» — false claim. The `ReadToEnd()` call is the first operation; in production (agent client → hook → JSON payload + EOF) it works; in manual testing without stdin pipe it hangs.
- **Resolution:** closed stdin explicitly with `echo '' | pwsh -File ...`, then hook completed correctly: exit 0, incremental re-index ran (chroma.sqlite3 rewritten, data_level0.bin grew 145→152 MB), health report written (4 areas: config/provider/index/mcp = pass).
- **Proposed fix (not yet applied, user chose to signal not fix now):** guard `if ([Console]::IsInputRedirected) { ... ReadToEnd ... }` to skip stdin read when not available.
- **Tracking:** promoted to backlog item FEAT-014 (Could) in epica E10 debito-tecnico; see `requirements/debito-tecnico/epic.md` (separate commit).
- **Lesson:** value of dogfooding — the defect emerged ONLY when the hook was executed for real, not in offline fixture tests (which use fake stdin paths). The feature born to prevent skipped mechanical steps had its own footgun, discovered by executing it.
