## [2026-06-21] record | FEAT-003 (MVP meccanico) — misura della fusione code+doc (fusion coverage)

Implementazione FEAT-003 (epica retrieval-qualita, branch 069-qualita-fusione-code-doc), pipeline SpecKit completa requirements→specify→plan→tasks→implement(meccanico). Estensione ADDITIVA dell'harness eval di FEAT-001/011 per misurare la **qualità della fusione code+doc**.

- **Il problema scoperto (dal commento dell'utente):** la riformulazione della mission ancorò il differenziatore di Sertor nella **fusione code+doc in un unico corpus**; le metriche hit@k/MRR/F1 misurano il *singolo strumento* (search_code / search_docs / find_symbol), ma NON **l'integrazione**. Un caso come «passare da requisito a implementazione» rimane nascosto se il top-k ha code senza doc o doc senza code — la fusion coverage lo rende **esplicito e contato**.

- **Vehicle CLI:** `sertor-rag eval run --fused` (switch opt-in, default OFF), `add-case --intent` / `amend-case --intent` per la genesi skill; nuovo modulo `services/eval/fusion.py` + `fused_runner.py`; factory `build_fused_eval_runner` in composition (Principio XI).
- **Evento osservabilità:** `fused_eval` metrics-only (case/coverage/intent_breakdown).
- **Estensione non-breaking:** `EvalReport`/`GroundTruth` invariati; `intent` è metadato dell'artefatto, non della firma API. A leve spente (OFF), comportamento identico al FEAT-001.
