## [2026-07-18] implement | E10-FEAT-033 implementata — ritual-check rileva il default branch (SpecKit completo)

Implementata E10-FEAT-033 «`ritual-check` rileva il default branch» via pipeline SpecKit completa, branch
`108-feat-033-ritual-check-default-branch`. 2° item della coda dell'analisi
[[setup-dichiara-presunzione-non-azione]] (dopo FEAT-038).

### Il bug e il fix

`_resolve_base` (`src/sertor_core/wiki_tools/ritual_check.py`) faceva `merge-base HEAD master` con `master`
**hardcodato** → su un repo con default `main` nessun merge-base → fail-loud `no merge-base with 'master'`.
Violazione Principio X (un assunto sull'ospite dentro un tool che legge tutto da config), e colpisce il
**primo** tool che un ospite tocca con la forced declaration del rituale (E10-FEAT-026). Segnalato da
**Noetix** (Acta), verificato sul codice.

**Fix:** nuovo helper `_default_base_candidates()` — rileva il default a runtime: (1) `origin/HEAD` via
`git symbolic-ref --short refs/remotes/origin/HEAD` (autorevole, il default *dichiarato dal remote*), poi
(2) i ref **esistenti** `origin/main`/`origin/master`/`main`/`master` (dedup, remoto prima di locale, `main`
prima di `master`). `_resolve_base` itera i candidati e ritorna il **primo con merge-base**; `--base`
invariato; fail-loud finale invariato (ora nomina i ref tentati). Il dogfood (`master`) è coperto dal path
primario (`origin/HEAD` del repo punta a `master`) → nessuna regressione.

### Decisioni di scope (forcella clarify, decisa in-flow)

Ordine di risoluzione = origin/HEAD → candidati ordinati esistenti → fail-loud (suggerimento Noetix,
sensato; nessun dubbio bloccante). Override del default in `wiki.config.toml` rinviato (Could).
