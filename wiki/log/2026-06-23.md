### FEAT-003 — primo run CI Windows: ROSSO → fix ermetico → verde (il payoff)

Pushato il branch + aperta **PR #96**; la CI Windows ha girato per la **prima volta in assoluto** su GitHub. **Esito: ROSSO** allo step «Tests — core (root)» — `3 failed, 974 passed` in `test_engine_selection.py` con `GloveUnavailableError [reason=cache_miss]` (+ ~3 min di hang).

**Causa-radice (test non-ermetico, NON regressione delle modifiche):** il default `glove` (FEAT-011) risolve la **cache GloVe in fase di costruzione** dell'embedder; in un checkout CI pulito non c'è cache → fail. In locale passava **solo** perché la macchina dev ha la GloVe in cache. Questi test verificano la **selezione del motore** (provider-agnostica), non l'embedding.

**Fix:** `tests/unit/test_engine_selection.py` — il helper `_settings` usa ora il provider **`hash`** (stdlib, zero-download), coerente con l'intento dichiarato del file («embedder costruito ma mai invocato: niente rete»). 8/8 verdi in 0.9s.

**Lezione:** è esattamente il valore di FEAT-003 — la CI su ambiente pulito coglie la non-ermeticità che il Windows locale (con cache) nasconde. Pattern da sorvegliare: ogni test che costruisce l'embedder di default tocca la risoluzione GloVe; deve forzare `hash` o iniettare un fake. *(Verifica «watch exit 0» fuorviante: lo stato reale del run va confermato con `gh run view`/`gh pr checks`, non col solo exit code del watch.)*
