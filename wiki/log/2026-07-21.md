## [2026-07-21] record | E4-FEAT-012 — la cattura-auto legge il gate dalla fonte reale

Delivered E4-FEAT-012 (merge 9921a34 / PR #210 on master, 2026-07-21). Closes the long-open auto-capture bug.

- **The bug:** the SessionEnd hook memory-capture gated on os.environ["SERTOR_MEMORY"], but that switch lives in .sertor/.env — loaded only by Settings.load inside the CLI, never injected into the hook process environment. So the gate always read None → silent no-op → auto-capture NEVER fired on any host enabling memory via the file. Manual sertor-rag memory archive worked (the CLI reads .env). Regression from the A-09 .ps1→.py migration.
- **Corrects a prior wrong hypothesis:** the EXEC previously attributed this bug to the relative-path cause of E10-FEAT-031/032. That was a second, stacked defect (hook script not found from a subfolder), now fixed — but underneath sat this os.environ gate, never touched. The empirical check disproved "self-closed": memory list still showed 2026-07-09 despite FEAT-031/032 installed.
- **The fix:** new _hooklib.memory_enabled() reads the same .env the CLI does (./.env then .sertor/.env anchored to CLAUDE_PROJECT_DIR; .env wins over os.environ, mirroring Settings.load), stdlib-only, byte-copied across both bundle copies (Claude/Copilot parity). +5 offline regression tests in test_portable_hooks_parity.py. Follow-up commit whitelisted SERTOR_MEMORY in test_host_agnostic.py.
- **Process lesson:** the root pytest -m "not cloud" (testpaths=["tests"]) does NOT collect packages/*/tests, so the host-agnostic failure only surfaced in CI. Run both suites locally before merge.
- **Verification:** full non-cloud suite 1214 passed; CI 8/8 green; sertor-core unchanged (only hook assets + tests). Proved LIVE on the dogfood: with os.environ lacking SERTOR_MEMORY, memory_enabled() → True from the real .sertor/.env; a manual archive filled the 2026-07-09→2026-07-20 gap (12 sessions). Post-merge: re-lock .sertor/ + incremental re-index + smoke MCP green.
- **Requirements:** requirements/memoria-conversazioni/feat-012-fix-cattura-auto-gate/. Also merged earlier this session: PR #209 (bookkeeping for FEAT-010).

## [2026-07-21] record | E4-FEAT-013 — ricerca semantica della memoria via MCP (memory_search semantic=true)

Delivered E4-FEAT-013 (merge `727ab5b` / PR #212 on master, 2026-07-21). Closes the "semantic via MCP" residue deferred from FEAT-010.

- **What:** MCP tool `memory_search` now accepts `semantic=true` → search past sessions BY MEANING (not keyword), mirroring the CLI `sertor-rag memory search --semantic`. Agents can now perform semantic recall over the native MCP surface instead of shelling out.
- **Design:** thin wrapper over the same core the CLI uses — `build_memory_semantic_index` + `SemanticMemoryQuery` + `index.search`. Behind the same two-layer gate as the CLI (`SERTOR_MEMORY` + `SERTOR_MEMORY_SEMANTIC`). When gates are off: `{"status":"disabled"}` naming the RIGHT knob, never silent fallback. Hit shape shared with full-text (session_key/captured_at/role/turn_index/snippet/score); query hashed in telemetry. Memoized builder with lru_cache.
- **Bug caught:** new helper `_hit_dict` for memory hits SHADOWED pre-existing `_hit_dict` for graph tools' `SymbolHit`, breaking `find_symbol`/`who_calls`/`get_context` (AttributeError). Full suite caught it; renamed memory helper to `_memory_hit_dict`. Lesson: run the full suite, not targeted subsets.
- **Verification:** +6 unit tests (semantic on/off gates, disabled knob naming, hit shape, query-not-in-clear, full-text default); full non-cloud suite green (1220 passed), ruff clean. Proved LIVE on real semantic index. Post-merge: re-lock `.sertor/` + incremental re-index (1406 doc) + smoke MCP green (`find_symbol("_semantic_search")` → `server.py:326`).
- **Code footprint:** `src/sertor_mcp/server.py` only; `sertor-core` INVARIANT. Requirements at [[memoria-conversazioni/feat-013-semantico-mcp]].
